# 도메인 문서

엔지니어링 스킬들이 코드베이스 탐색 시 이 저장소의 도메인 문서를 어떻게 소비해야 하는지 정의합니다.

## 탐색 전 읽어야 할 파일

- 저장소 루트의 **`CONTEXT.md`**, 또는
- 루트에 **`CONTEXT-MAP.md`** 가 있다면 — 컨텍스트별 `CONTEXT.md` 위치를 가리킵니다. 주제와 관련된 항목을 모두 읽으세요.
- **`docs/adr/`** — 작업하려는 영역과 관련된 ADR을 읽으세요. 다중 컨텍스트 저장소에서는 컨텍스트별 결정을 위해 `src/<context>/docs/adr/` 도 함께 확인합니다.

이 파일들이 없다면 **조용히 진행하세요**. 부재를 지적하거나 사전에 만들 것을 제안하지 마세요. 생성 스킬(`/grill-with-docs`)이 용어나 결정이 실제로 결정될 때 lazy하게 만듭니다.

## 파일 구조

단일 컨텍스트 저장소 (대부분의 경우):

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-event-sourced-orders.md
│   └── 0002-postgres-for-write-model.md
└── src/
```

다중 컨텍스트 저장소 (루트에 `CONTEXT-MAP.md` 가 있는 경우):

```
/
├── CONTEXT-MAP.md
├── docs/adr/                          ← 시스템 전역 결정
└── src/
    ├── ordering/
    │   ├── CONTEXT.md
    │   └── docs/adr/                  ← 컨텍스트별 결정
    └── billing/
        ├── CONTEXT.md
        └── docs/adr/
```

## 글로서리의 어휘를 사용하세요

출력에서 도메인 개념을 명명할 때 (이슈 제목, 리팩터링 제안, 가설, 테스트 이름 등) `CONTEXT.md` 에 정의된 용어를 그대로 사용하세요. 글로서리가 명시적으로 피하는 동의어로 표류하지 마세요.

필요한 개념이 아직 글로서리에 없다면 그것은 신호입니다 — 프로젝트가 사용하지 않는 언어를 만들어내고 있거나(재고하세요), 실제로 갭이 있다는 뜻이니(`/grill-with-docs` 용으로 기록).
